# Cross-Platform Distribution And Installer Standard

This standard applies to **applications built with AIAST**, not to the AIAST
template lifecycle itself. For AIAST copy/update/repair semantics, see
`INSTALLER_AND_UPGRADE_CONTRACT.md`.

**Agent operational checklist:** For binding rules on *when* to scaffold
installers early, *how* to validate production-like host installs (including
desktop icons), governed ports, dependency/DB setup, and **post-work launch/render
verification**, read `AGENT_INSTALLER_AND_HOST_VALIDATION_PROTOCOL.md`.

## Goals

- Every shipping app defines **explicit, automatable** install and uninstall
  paths per target platform.
- Installers minimize post-install manual work: only **elevation**, **secrets
  the app cannot derive** (for example API keys), and **irreversible data
  destruction** require interactive confirmation.
- **Network bindings** avoid collisions with other apps through governed port
  ranges, persisted assignments, and documented override flags. Prefer
  `registry/port_assignments.yaml`, `tools/port_allocator.py`, and
  `preflight_port_scan.py` from generated runtime foundations over ad hoc
  ports.
- **Hardening** is default-on: least privilege, loopback-first services,
  non-wildcard binds in examples, health checks on composed backends, and
  alignment with `SECURITY_HARDENING_CONTRACT.md`.

## Minimum launch milestone (early installer scaffolds)

Create or refresh **install and distribution scaffolds as soon as the app is
first renderable and launchable** on a developer machine (for example a working
`npm run dev`, `cargo run`, or equivalent), **even if the product is not feature
complete**. Goal: the builder can **dogfood a user-like install path** on a host
(compose, `ops/install`, `distribution/platforms/*`, packaging notes) while
iteration continues. Refine those surfaces as behavior stabilizes; do not wait
for “release polish” before the first end-to-end install smoke.

## Directory layout (project-owned after scaffold)

Place platform-specific delivery assets under **`distribution/`** at the repo
root (generated by `bootstrap/generate-runtime-foundations.sh`). Do not mix
unrelated platform files in a single flat directory.

```
distribution/
  README.md
  platforms/
    linux/      # wraps or documents ops/install, systemd, packaging
    windows/    # PowerShell or MSI/chocolatey/winget notes + scripts
    macos/      # pkg, Homebrew formula stub, notarization notes
    android/    # Gradle/Flutter; links to mobile/ when applicable
    ios/        # Xcode, TestFlight, enterprise distribution notes
```

Linux desktop packages (`Flatpak`, `Snap`, `AppImage`) remain under
`packaging/` as today; `distribution/platforms/linux/` explains how those
artifacts relate to `ops/install/*.sh`.

## Required installer capabilities (menu)

Implement as many of the following as the product shape allows. Name flags
consistently across shells where possible.

| Operation | Intent |
|-----------|--------|
| **install** | First-time placement, dependency satisfaction, config generation, service registration (if any). |
| **upgrade** | Replace binaries/assets; migrate config and data; re-run validation. |
| **repair** | Restore missing files, units, or permissions without deleting user data. |
| **uninstall** | Remove app binaries, services, and desktop integration; optional data retention prompt. |
| **purge** | Destructive removal including config and local databases after explicit confirm. |
| **reset-data** | Truncate or recreate application schema while keeping install layout. |
| **reconfigure** | Re-run bind/port/URL discovery and persist new values. |
| **status** / **doctor** | Report versions, listeners, and health without mutating state. |

Linux reference implementation: `ops/install/install.sh`, `repair.sh`,
`uninstall.sh`, `purge.sh`.

## OS and architecture

- **Detect** OS family, version, and CPU architecture before choosing paths or
  packages.
- **Linux**: support `x86_64`, `aarch64`, and (when relevant) `armv7l` via
  distro packages, static binaries, or containers; document musl vs glibc where
  it matters.
- **Windows**: `x64`, `arm64`; prefer signed installers or package managers for
  distribution.
- **macOS**: Apple Silicon and Intel; document Gatekeeper and notarization
  expectations.
- **Android / iOS**: treat as **separate deliverables** (APK/AAB and App Store /
  TestFlight). Installers are build-and-deploy pipelines, not a single shell
  script. Cross-link `mobile/` and `MOBILE_GUIDE.md`.

## Databases, web servers, and dependencies

- Prefer **declared** dependencies in compose, packages, or lockfiles—not
  silent global installs—unless the product is explicitly an appliance.
- If the installer provisions PostgreSQL, Redis, or similar, use **version-pinned**
  images or distro packages, **non-published** compose ports by default, and
  documented admin credentials or socket auth.
- Web servers and reverse proxies should terminate TLS at the edge; app
  listeners stay on loopback unless the threat model requires otherwise.

## Error handling and logging

- Fail fast with **actionable** messages; exit codes documented in each script’s
  header.
- Log to an app-owned path under the install prefix or user config dir; never
  log secrets.
- Support **dry-run** where feasible for filesystem and service changes.

## Validation

- CI must build or dry-run packaging/install paths for each claimed platform.
- Run `bootstrap/check-runtime-foundations.sh` and
  `bootstrap/check-network-bindings.sh` after changing install or port surfaces.
